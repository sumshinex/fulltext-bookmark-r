# fulltext-bookmark-main WebDAV 单向备份与更大 Popup 设计

- 日期：2026-04-11
- 主题：为全文书签扩展新增 WebDAV 单向备份/恢复能力，并将扩展 Popup 固定放大
- 设计状态：已确认，待实现规划

## 1. 背景与目标

当前项目已有两类相关能力，但都不能直接满足本次需求：

1. 现有远程发送能力 `remoteStore` 只是在后台向一个 URL 发送页面元数据，语义更接近“远程推送”，不是备份系统。
2. 现有导入导出能力主要面向 CSV，适合人工查看和轻量迁移，不适合作为全文数据的可靠备份载体。
3. 当前 Popup 搜索页和 Ask GPT 页尺寸偏窄，长标题、长 URL、较长回答的可读性一般。

本次目标是：

- 新增 **WebDAV 单向备份/恢复** 能力
- 备份载体采用 **单个结构化 JSON 文件**，覆盖全文搜索恢复所需的本地核心数据
- WebDAV 能力仅放在设置页，不进入 Popup
- 增加连接测试、最后备份时间展示、恢复确认等增强交互
- 将扩展 Popup 统一固定放大，但仍保持浏览器扩展 Popup 形态

## 2. 约束与明确决策

本设计基于以下已确认约束：

1. WebDAV 只做 **单向备份/恢复**，不做双向同步。
2. 当前用户选择的是 **单向备份模型 A**：上传完整备份，后续由用户手动恢复。
3. 本次不复用 CSV 作为 WebDAV 主备份格式；CSV 继续保留为现有人工导入导出能力。
4. 本次不重构本地运行时存储，不把本地数据改造成可自定义路径的单文件数据库。
5. 浏览器扩展环境下，本地运行时数据仍由浏览器存储系统管理，不能让用户自定义数据库文件位置。
6. WebDAV 入口只放设置页，Popup 不增加备份入口。
7. Popup 放大采用 **固定更宽更高尺寸**，不改为独立窗口，不做复杂自适应方案。
8. 恢复策略采用 **覆盖恢复**，不做合并恢复、冲突处理或局部恢复。
9. 本次增强体验包括：测试连接、最近备份状态、最后备份时间、恢复确认。
10. 本次不做自动定时备份，不做历史版本轮转。

## 3. 方案选择

### 3.1 备选方案

#### 方案 A：最小改动方案
- 设置页新增 WebDAV 配置与手动备份/恢复
- 备份载体为结构化 JSON
- Popup 统一放大

**优点**
- 改动最小
- 风险最低
- 最快落地

**缺点**
- 缺少连接测试、状态反馈等增强体验

#### 方案 B：增强备份方案（推荐并已确认）
- 在方案 A 基础上增加连接测试、最后备份时间、最近操作状态、覆盖确认
- 保持手动备份/恢复，不做自动任务
- Popup 统一固定放大

**优点**
- 体验完整度更高
- 仍能控制实现复杂度
- 与当前项目体量匹配

**缺点**
- 状态管理和错误处理比最小方案更多

#### 方案 C：CSV 复用方案
- WebDAV 上传/下载 CSV
- 恢复时继续走现有 CSV 导入

**优点**
- 看似复用现有逻辑较多
- 文件便于人工查看

**缺点**
- 不适合高保真恢复全文数据
- 不利于后续字段扩展
- 备份/恢复语义不够稳定

### 3.2 结论

采用 **方案 B：增强备份方案**。

即：
- WebDAV 使用结构化 JSON 单文件备份
- 设置页新增独立的 WebDAV Backup 区块
- 提供测试连接 / 备份 / 恢复 / 状态展示
- Popup 采用统一固定尺寸放大

## 4. 信息架构设计

### 4.1 设置页结构调整

本次不把 WebDAV 混入现有 `remoteStore` 设置，而是新增一个独立区域：

- Remote API Settings（现有）
- Data Management（现有 CSV 导入导出）
- **WebDAV Backup（新增）**

这样可以明确区分三类语义：

- `remoteStore`：远程推送页面数据
- CSV：人工导入导出
- WebDAV Backup：完整备份与恢复

### 4.2 WebDAV Backup 区块字段

建议字段如下：

- WebDAV 基础目录 URL
- 用户名
- 密码 / App Password
- 备份文件名
- 最后备份时间
- 最近一次操作状态（成功 / 失败摘要）

说明：
- URL 字段表示 WebDAV 根路径或备份目录，不直接包含最终文件名
- 文件名默认为 `fulltext-bookmark-backup.json`
- 密码字段以掩码形式输入
- 配置不完整时，备份/恢复/测试按钮应禁用或给出明确提示

### 4.3 WebDAV Backup 区块动作

建议动作如下：

- **测试连接**
- **备份到 WebDAV**
- **从 WebDAV 恢复**

交互规则：
- 测试连接：不改本地数据，也不更新最后备份时间
- 备份到 WebDAV：成功后更新最后备份时间和最近状态
- 从 WebDAV 恢复：必须先弹确认，再执行覆盖恢复

## 5. 数据结构与备份文件设计

### 5.1 备份载体

采用 **单个结构化 JSON 文件** 作为 WebDAV 备份文件。

建议包含：

- `schemaVersion`
- `exportedAt`
- `source`
- 数据表内容
  - `pages`
  - `contents`
  - 以及恢复全文搜索所需的其他本地核心表
- 可选统计信息
  - 各表记录数
  - 扩展版本信息

### 5.2 为什么不用 CSV

CSV 适合：
- 人工查看
- Excel 打开
- 轻量迁移

CSV 不适合：
- 高保真恢复
- 保留表结构与字段扩展空间
- 稳定承载全文搜索数据的备份/恢复

因此：
- **CSV 保留**，但不参与 WebDAV 主链路
- **WebDAV 只使用结构化 JSON**

### 5.3 本地存储策略

本地运行时存储维持现状：

- UI / 配置状态继续走扩展存储与 redux persist
- 页面数据继续走 Dexie / IndexedDB

本次不把本地运行时存储重构为可自定义路径的单文件数据库。

原因：
- 浏览器扩展环境本身不适合让用户指定运行时数据库位置
- 该重构范围明显超出本次目标
- 本次真正需要的是“单文件备份”，而不是“单文件运行时存储”

## 6. 后台命令与职责设计

后台负责 WebDAV 网络交互、备份文件生成与恢复控制。

建议新增命令：

- `webdav_test_connection`
- `webdav_backup_export`
- `webdav_backup_restore`

### 6.1 `webdav_test_connection`
职责：
- 验证 URL 是否可达
- 验证认证是否有效
- 验证目标路径或父目录是否可访问

返回：
- `ok`
- 失败原因摘要

### 6.2 `webdav_backup_export`
职责：
- 从本地数据库导出完整结构化数据
- 生成单个 JSON 备份内容
- 通过 WebDAV `PUT` 上传到目标文件
- 成功后返回备份时间和结果摘要

### 6.3 `webdav_backup_restore`
职责：
- 从 WebDAV 下载备份文件
- 校验 JSON 结构和版本
- 清理本地相关表
- 将备份导入本地
- 返回恢复结果摘要

### 6.4 职责边界

前端负责：
- 配置录入
- 按钮交互
- 状态展示
- 确认弹窗

后台负责：
- 网络请求
- 认证头处理
- 文件上传/下载
- 结构校验
- 恢复执行
- 错误归一化

## 7. WebDAV 交互设计

### 7.1 认证方式

采用标准 WebDAV 基本认证（用户名 + 密码 / App Password）。

请求层面：
- 使用 `Authorization: Basic ...`
- 不在日志、错误提示、测试数据中暴露明文密码

### 7.2 文件路径策略

用户提供的 WebDAV URL 视为完整目标文件路径或目标目录的基础路径。

推荐设计：
- URL 表达 WebDAV 根路径或备份目录
- 文件名由单独字段指定
- 最终目标路径由后台组合

这样更便于：
- 更换文件名
- 后续扩展默认文件策略
- 避免把文件名硬编码在 URL 文本里

### 7.3 默认文件名

默认：
- `fulltext-bookmark-backup.json`

本次不引入：
- 自动时间戳文件名
- 多版本目录轮换
- 最近若干备份保留策略

## 8. 备份与恢复流程设计

### 8.1 备份流程

1. 用户在设置页点击“备份到 WebDAV”
2. 前端检查配置完整性
3. 调用后台导出命令
4. 后台导出本地数据并上传到 WebDAV
5. 成功后前端更新：
   - 最后备份时间
   - 最近一次操作结果
6. 前端给出成功提示

### 8.2 恢复流程

1. 用户点击“从 WebDAV 恢复”
2. 前端弹出覆盖确认
3. 后台下载备份文件
4. 后台校验结构和版本
5. 后台清理本地相关表
6. 后台导入备份数据
7. 前端展示恢复结果
8. 提示用户重新打开 Popup 或刷新扩展页面

### 8.3 覆盖确认文案要求

恢复前必须明确告知：
- 此操作会覆盖当前本地数据
- 建议先手动导出当前本地备份
- 恢复失败不能默默吞错

## 9. 错误处理设计

### 9.1 测试连接失败

错误应尽量分型返回：

- URL 非法
- 网络不可达 / 超时
- 401 / 403 认证失败
- 目标路径不可写
- 目标文件不存在（恢复场景）
- WebDAV 响应不符合预期

### 9.2 备份失败

规则：
- 不更新最后备份时间
- 保留最近一次失败状态
- 不影响本地现有数据

### 9.3 恢复失败

规则：
- 下载失败：中止，不改本地
- 结构校验失败：中止，不改本地
- 导入失败：返回明确错误，不允许静默失败

### 9.4 错误文案原则

错误信息需要：
- 具体
- 可定位
- 不泄露认证敏感信息

例如：
- “认证失败，请检查用户名或密码”
- “目标路径不可写，请检查 WebDAV 目录权限”
- “备份文件不存在，无法恢复”

## 10. Popup 放大设计

### 10.1 目标

Popup 放大应解决以下问题：
- 搜索结果列表更易读
- 长标题 / URL 展示更自然
- Ask GPT 回答区和来源区不再过于拥挤

### 10.2 方案

采用 **统一固定尺寸放大**：
- Popup 主容器更宽、更高
- SearchView 跟随新的统一尺寸
- GPTSearch 跟随新的统一尺寸
- 内部结果区继续滚动

### 10.3 不做的事项

本次不做：
- 独立窗口模式
- 按 tab 切换不同尺寸
- 复杂响应式自适应布局
- 结构性重写 Popup 交互

### 10.4 预期效果

- 搜索 tab：结果卡片更容易浏览
- Ask GPT tab：回答区与来源区更清晰
- Settings 跳转入口继续保留现状

## 11. 验证方案

### 11.1 WebDAV 验证

最小必要验证包括：

1. 测试连接成功
2. 错误认证时返回明确错误
3. 成功导出并上传备份文件
4. 成功从 WebDAV 下载并恢复
5. 恢复后本地搜索数据可正常使用

### 11.2 Popup 验证

最小必要验证包括：

1. 搜索页按新尺寸正常显示
2. Ask GPT 页按新尺寸正常显示
3. 列表区域滚动正常
4. 无明显裁切、tab 挤压、按钮错位

### 11.3 回归验证

至少回归以下能力：
- 现有 CSV 导入导出仍可用
- 现有 `remoteStore` 不受影响
- Ask GPT 请求链路仍正常
- 普通搜索 Popup 仍正常展示结果
- 类型检查通过

## 12. 分阶段实现建议

建议实现顺序：

1. 扩展本地状态模型，新增 WebDAV 配置与状态字段
2. 新增设置页 WebDAV Backup 区块
3. 新增后台 WebDAV 测试命令
4. 新增后台备份导出与上传命令
5. 新增后台下载与恢复命令
6. 增加恢复确认与状态展示
7. 放大 Popup 尺寸并微调内部滚动区域
8. 执行 WebDAV / Popup / 回归验证

## 13. 非目标

以下内容明确不在本次范围内：

- 双向同步
- 自动定时备份
- 多版本备份历史
- 冲突处理
- 局部恢复
- 自定义本地数据库路径
- 本地运行时数据库重构为单文件数据库
- Popup 中增加 WebDAV 快捷入口
